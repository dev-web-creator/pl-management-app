# 開発プロセスと進め方

このドキュメントは「どういう順序で開発しているか」「手戻りする場合どこをどう変えるか」を
追えるようにするためのものです。拡張・保守の起点として常に最新に保ちます。

## 開発フェーズ（全体ロードマップ）

| Phase | 名称 | 目的 | 成果物 | 状態 |
|-------|------|------|--------|------|
| 0 | 要件のたたき台 | コンセプト・要件の言語化 | `docs/requirements.md` | ✅ 完了 |
| 1 | **UI/UXモックアップ** | 画面と操作感の確認（捨てやすいHTML） | `mockup/index.html` | 🚧 進行中 |
| 2 | 現運用のすり合わせ | ユーザーの実運用詳細を反映し要件更新 | requirements.md 更新 / `docs/master-data.md` / ADR-001〜026 | ✅ 完了 |
| 3 | 基本設計 | 確定したER図・画面遷移・API一覧 | `docs/database-design.md` 他 | 🚧 ER完了。画面遷移/API一覧が残 |
| 4 | 実装（DB→API→UI） | ローカルPostgreSQL + Next.js | アプリ本体 | 🚧 DDL/seed作成（`db/`）。検証待ち |
| 5 | AWS移行・本番化 | RDS等へのデプロイ | インフラ定義 | ⬜ 未着手 |

> 詳細設計（クラス設計レベル）は Phase 3 の基本設計に内包し、独立フェーズにはしない方針。

## 進め方のルール（手戻りを追えるようにする仕組み）

1. **ドキュメントを「正」とする**: コードより先に `docs/` を更新する。実装はドキュメントに従う。
2. **意思決定はすべて `docs/decisions.md` に記録する**（ADRライト）。
   「いつ・何を・なぜそう決めたか」を残し、後から覆す時は新しいエントリで上書き履歴を残す。
3. **手戻り時のルール**: 要件が変わったら
   `requirements.md` → `decisions.md` → 設計ドキュメント → コード の順に上流から直す。
   どのフェーズに戻ったかをこの表の「状態」欄で示す。
4. **MVP優先**: まず動く最小単位を作り、運用しながら育てる。
