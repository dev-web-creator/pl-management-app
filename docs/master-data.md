# マスタデータ定義（現運用の整理）

ユーザーの現運用（スプレッドシート）をシステムのマスタデータとして整理したもの。
🔶 = 要確認ポイント（`docs/decisions.md` で確定後に反映）。
このファイルは将来 DB の初期データ（seed）の元になる。

---

## 0. タブ構成（現運用の全体像）

| タブ | 役割 | 入力方法 |
|------|------|----------|
| 日次（支出） | 変動費のみを1円単位で日次入力 | 手入力 |
| 月次（支出） | 固定費＋変動費を月次集計。FY（4月〜3月）で振り返り | 固定費=手入力 / 変動費=日次から自動集計 |
| 月次（収入） | 給与・控除・各種収入を月次管理 | 給与明細OCR＋手入力 |

- **会計年度（FY）= 4月〜翌3月**。年間でPLを振り返れることを重視。
- 固定費は日次入力せず、月次タブで「定額を12か月ぶん仮置き → 解約したら0円＋グレーアウト」運用。
  - 🔶 システムでは「固定費マスタ（金額・開始/終了年月）」で再現する想定（ADR-003系）。「0円グレーアウト」＝終了年月のセット。

## 1. 変動費カテゴリ階層（PL区分: variable_cost）

3〜4階層。**取引入力は最下層（葉）に対して行い、上位は自動集計**。

```
変動費
├─ 食費
│   ├─ 食費(1人)          ← 自動集計（朝+昼+晩）。入力対象外
│   │   ├─ 朝飯           ★日次入力
│   │   ├─ 昼飯           ★日次入力
│   │   └─ 晩飯           ★日次入力
│   ├─ 交際費             ★日次入力（誰かと飲み・ごはん。旅行中含む／＝会食）
│   ├─ プレゼント・奢り    ★日次入力
│   └─ スーパー・まとめ買い ★日次入力（Amazon含む）
├─ 自己投資・1人体験
│   ├─ 1人体験            ★日次入力（運動・散髪・グリーン車・1人映画 等）
│   └─ 結婚式             ★日次入力
└─ 諸経費・備品購入費
    ├─ 旅費・交通費        ★日次入力（交通系IC・新幹線代）
    ├─ 物品購入費          ★日次入力（生活用品・贅沢品など物品全般）
    ├─ その他諸経費        ★日次入力（引越し費・処分費・手数料 等）
    └─ 嗜好品(TBC)         ★日次入力
```

- ✅ グループ名は「食費」、`食費(1人)`＝会食以外の自分の食事、`交際費`＝会食。交際費は食費グループ所属（確定）。
- 🔶 `④食費合計`（日次タブ）＝ `食費(1人)`（月次タブ）と同一の自動集計、という理解でよいか。

## 3. 収入カテゴリ（月次収入タブ）

「月次合計（可処分所得）」配下。
🔶 用語: ここでの「可処分所得」は"収入の合計"を指す。一方ダッシュボードの最終利益（＝収入−支出）も可処分所得と呼んでいる。表記の整理が必要。

| 項目 | PL上の扱い（要確認） |
|------|----------------------|
| 給与収入（控除後手取り） | 収入（income） |
| 総支給額（給与） | 内訳の親（手取り＋控除の合計） |
| 経費精算 | **PL対象外（excluded）**: 立替の戻り。残高には反映、PL集計から除外 |
| 家族収入 | 収入（income） |
| その他収益 | 収入（income） |
| ポイント収入 | 収入（income） |
| 投資収益（売却益・配当） | 収入（income）。実現益のみ手入力で計上（ADR-014） |
| 借入金 | **PL対象外（excluded）**: 負債。残高には反映、PL集計から除外 |

> **PL区分（pl_type）の値**: `income` / `fixed_cost` / `variable_cost` / `deduction` / `excluded`
> `excluded` は「現金は動くがPL（損益）には載せない」項目（経費立替・精算・借入金・元金返済など）。
> 記録は残すので"片方だけ入力"を防げ、かつPLの比率（変動費率・貯蓄率）が歪まない。
>
> **可処分所得（トップライン）の定義**: 全収入合計（給与手取り＋副業その他＋家族＋ポイント＋投資収益）。
> 経費精算・借入金は `excluded` なので含めない（現運用シートは含めるが、本アプリは除外＝PL精度向上 / ADR-010・013）。
>
> **借入金まわり（ADR-021）**: 借入受取・元金返済＝excluded（負債の増減）。**返済利子＝支払い利息（PL費用）**。
> **税金・社保（ADR-022）**: 控除内訳（deduction）で管理。支出には計上しない（手取りに織り込み済み・二重計上防止）。

## 4. 控除内訳（PL区分: deduction）

総支給額から手取りを出すための控除。還付項目はマイナス控除（＝戻し）。

| 項目 | 符号 |
|------|------|
| 住民税 | − 控除 |
| 所得税 | − 控除 |
| 健康保険料 | − 控除 |
| 雇用保険料 | − 控除 |
| 厚生年金保険料 | − 控除 |
| 子ども・子育て支援金 | − 控除 |
| 定額減税・確定申告還付 | ＋ 戻し |
| 年末調整精算額 | ± どちらもあり得る |
| ふるさと納税返金額 | ＋ 戻し |
| 総支給額に対する比率 | 算出値（控除合計 ÷ 総支給額） |

## 5. 給与内訳（給与明細の構成・KPI）

金額項目と、労働時間・時給換算などの**非金額のKPI**が混在。

**金額項目**: 月給 / 固定時間外手当(30h固定) / 固定時間外手当(30h超過分) / 時間外労働手当(60h超過・追加割増) / 深夜労働手当 / 勤怠控除 / 非課税通勤手当 / 課税通勤手当 / 残業代合計

**KPI（金額ではない・多くは算出値）**: 総労働時間 / 時間外労働時間 / 総労働時間の時給換算 / 残業代の時給換算

- 🔶 KPI（労働時間・時給換算）をアプリの管理対象にするか。するなら payslip に時間項目を持たせ、時給換算は算出で出す。

## 6. ウォレット（口座・カード）マスタ

### 銀行・電子マネー
| ウォレット | 種別 | 主な役割 |
|------------|------|----------|
| みずほ銀行 | bank | **本業の給与振込先**、家賃の引き落とし、iDeCoの引き落とし |
| 三井住友銀行 | bank | 副業/その他収益の振込先、**全カードの引き落とし口座**、楽天証券への入金（株購入）時の出金 |
| 三菱UFJ銀行 | bank | 現在は休眠（過去利用）。将来の分散用に残す |
| 楽天銀行 | bank | 楽天証券と連携。株関連の入出金が多い |
| PayPay残高 | prepaid | PayPay残高払いの出金元。実残高と一致させたい |
| ANA Pay | prepaid | 点活用の中継。カードからチャージ |
| JAL Pay | prepaid | 点活用の中継（JAL Pay→ANA Pay→PASMO の連鎖あり） |
| PASMO | prepaid | 交通系。チャージして交通費等に使用 |
| ICOCA | prepaid | 交通系 |
| VポイントPay | prepaid | |
| メルペイ | prepaid | |
| Pontaポイント | points | |
| dポイント | points | |
| PayPayポイント（自動運用）| points | 🔶 自動運用で変動。ポイント扱い/投資扱いか要確認 |

> 全プリペイドを残高管理（ADR-024）。チャージ＝資金移動（transfer）。
> ポイントは資産（残高管理）として扱い、取引は満額＋分割支払い（split tender）で表現（ADR-025）。
> 銀行は支店情報も保持：みずほ(浜松町)/三菱UFJ(国立)/三井住友(オリーブECRU 871)/楽天(ベース)。
> ※ウォレットの最終確定リストはseed時に。資産合計には銀行＋プリペイド＋ポイントを含む（ADR-026）。

- **現金（確定 ADR-015）**: 95%キャッシュレス。`現金`は決済手段として選択可・支出は計上するが、**残高は管理対象外**（追わない）。
- **PayPayカード払いはクレカ側に記録／PayPay残高払いは PayPay残高ウォレットに記録**（混同しない）。
- **投資の扱い（確定 ADR-014）**: 株購入・iDeCo拠出など元本移動はPL非計上（excluded／証券への振替）。
  含み益は追わない。**実現益・配当のみ手入力で「投資収益」(income)に計上**。証券/iDeCo残高管理はMVPスコープ外（将来）。

### 固定費の引き落とし口座（項目ごとに異なる点に注意）
